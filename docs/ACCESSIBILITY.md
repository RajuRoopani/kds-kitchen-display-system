# Kitchen Display System (KDS) — Accessibility Checklist\n\n**Compliance Target:** WCAG 2.1 Level AA  \n**Last Updated:** 2024  \n**Status:** Phase 2 Implementation\n\n---\n\n## Overview\n\nThis document defines the accessibility (a11y) requirements and verification checklist for the KDS application. All interactive components must be keyboard navigable, screen-reader compatible, and meet color contrast standards.\n\n---\n\n## WCAG 2.1 AA Compliance Checklist\n\n### 1. Keyboard Navigation (WCAG 2.1.1 Keyboard)\n\n**Requirement:** All functionality must be accessible via keyboard alone.\n\n#### Checklist Items\n\n- [ ] **Tab order is logical**\n  - Tab through all interactive elements (buttons, cards, tabs)\n  - Order should follow visual layout: left-to-right, top-to-bottom\n  - Test: `Tab` key cycles through all interactive elements\n\n- [ ] **Tab through order cards**\n  - Each card is focusable (tabindex=\"0\" or semantic <button>)\n  - Focus visible (outline or highlight)\n  - Enter/Space key activates card (opens modal)\n  - Test: `Tab` to card, press `Enter` → modal opens\n\n- [ ] **Tab through action buttons**\n  - Accept, Ready, Complete, Cancel buttons are focusable\n  - Button state (enabled/disabled) is visually clear\n  - Enter/Space key activates button\n  - Test: `Tab` to button, press `Enter` → action dispatches\n\n- [ ] **Tab through modal elements**\n  - Close button, order details, action buttons all focusable\n  - Focus wraps within modal (no focus escape)\n  - Test: `Tab` cycles through modal elements\n\n- [ ] **Tab through tab selector (mobile)**\n  - Each tab is focusable\n  - Left/Right arrow keys switch tabs (optional but recommended)\n  - Enter/Space key activates tab\n  - Test: `Tab` to tab, press `Enter` → column switches\n\n- [ ] **Escape key closes modal**\n  - Modal must be closeable without mouse\n  - Test: Open modal, press `Esc` → modal closes\n\n- [ ] **No keyboard trap**\n  - User can always exit any component with Tab/Escape\n  - No focus locked in infinite loop\n  - Test: Press Tab repeatedly, ensure navigation continues\n\n#### Implementation\n\nUse semantic HTML:\n```tsx\n// ✓ Good: semantic <button> (built-in keyboard support)\n<button onClick={handleAccept}>Accept</button>\n\n// ✓ Good: non-semantic element with explicit keyboard support\n<div\n  role=\"button\"\n  tabIndex={0}\n  onKeyPress={(e) => {\n    if (e.key === 'Enter' || e.key === ' ') handleAccept();\n  }}\n>\n  Accept\n</div>\n\n// ✗ Bad: no keyboard support\n<div onClick={handleAccept}>Accept</div>\n```\n\n---\n\n### 2. Focus Indicators (WCAG 2.4.7 Focus Visible)\n\n**Requirement:** All interactive elements must have a visible focus indicator.\n\n#### Checklist Items\n\n- [ ] **Order cards have visible focus ring**\n  - Focus state: 2px solid outline, color #2196f3\n  - Offset: 2px from element edge\n  - Contrast ratio: ≥3:1 against background\n  - Test: Tab to card, verify outline visible\n\n- [ ] **Buttons have visible focus ring**\n  - Focus state: 2px solid outline\n  - Color: contrasts with button background\n  - Test: Tab to button, verify outline visible\n\n- [ ] **Tabs have visible focus ring (mobile)**\n  - Focus state: 2px solid outline or background change\n  - Test: Tab to tab, verify focus visible\n\n- [ ] **Modal close button has visible focus**\n  - Focus state: clearly distinguishable\n  - Test: Tab to close button, verify visible\n\n- [ ] **No focus hidden with CSS**\n  - Do NOT use `outline: none` without replacement\n  - Always provide visible alternative\n\n#### Implementation\n\n```css\n/* ✓ Good: Custom focus style */\nbutton:focus {\n  outline: 2px solid #2196f3;\n  outline-offset: 2px;\n}\n\n/* ✗ Bad: Hidden focus */\nbutton:focus {\n  outline: none;\n}\n```\n\n---\n\n### 3. Color Contrast (WCAG 1.4.3 Contrast Minimum)\n\n**Requirement:** All text must have sufficient contrast ratio (4.5:1 for body text, 3:1 for UI components).\n\n#### Checklist Items\n\n- [ ] **Order ID contrast**\n  - Text: #1a1a1a (dark gray), Background: #ffffff (white)\n  - Ratio: 16.43:1 ✓ (exceeds 4.5:1)\n  - Test: Use WebAIM Contrast Checker\n\n- [ ] **Customer name contrast**\n  - Text: #555 (medium gray), Background: #ffffff\n  - Ratio: 8.6:1 ✓ (exceeds 4.5:1)\n  - Test: Verify ratio\n\n- [ ] **Items list contrast**\n  - Text: #999 (light gray), Background: #ffffff\n  - Ratio: 4.5:1 ✓ (meets 4.5:1)\n  - Test: Verify ratio\n\n- [ ] **Timestamp contrast**\n  - Text: #999, Background: #ffffff\n  - Ratio: 4.5:1 ✓\n  - Test: Verify ratio\n\n- [ ] **Button text contrast**\n  - Enabled: white text on #2196f3\n  - Ratio: 8.6:1 ✓\n  - Disabled: #999 text on #e0e0e0\n  - Ratio: 3.2:1 (check meets 3:1 for UI components)\n  - Test: Verify both states\n\n- [ ] **Error banner contrast**\n  - Text: #856404 (dark brown), Background: #fff3cd (light yellow)\n  - Ratio: 8.5:1 ✓\n  - Test: Verify ratio\n\n- [ ] **Tab selector contrast**\n  - Selected: white text on #2196f3\n  - Ratio: 8.6:1 ✓\n  - Unselected: #666 on #f5f5f5\n  - Ratio: 5.4:1 ✓\n  - Test: Verify both states\n\n- [ ] **Order count badge contrast**\n  - Text: #666 (gray), Background: #f5f5f5 (light gray)\n  - Ratio: 5.4:1 ✓\n  - Test: Verify ratio\n\n- [ ] **No color as only means of information**\n  - Status badges must have text labels, not just colors\n  - Example: \"Status: Received\" (not just a colored box)\n  - Test: Verify status is clear without color\n\n#### Testing Tools\n\n- **WebAIM Contrast Checker:** https://webaim.org/resources/contrastchecker/\n- **axe DevTools:** Chrome extension for automated checks\n- **WAVE:** Web Accessibility Evaluation Tool\n\n#### Color Palette (Verified Contrasts)\n\n| Color | Hex | Text Color | Ratio | Usage |\n|-------|-----|-----------|-------|-------|\n| Primary | #2196f3 | #ffffff | 8.6:1 | Buttons, tabs (active) |\n| Destructive | #f44336 | #ffffff | 5.9:1 | Cancel button, errors |\n| Success | #4caf50 | #ffffff | 5.2:1 | Success messages |\n| Warning | #fff3cd | #856404 | 8.5:1 | Error banner |\n| Light gray | #f5f5f5 | #666 | 5.4:1 | Backgrounds, badges |\n| Dark text | #1a1a1a | #ffffff | 16.4:1 | Headings, IDs |\n| Body text | #555 | #ffffff | 8.6:1 | Customer names, items |\n| Muted text | #999 | #ffffff | 4.5:1 | Timestamps, labels |\n\n---\n\n### 4. ARIA Labels & Semantic HTML (WCAG 2.1.1, 4.1.2)\n\n**Requirement:** All interactive elements must have accessible names/labels.\n\n#### Checklist Items\n\n- [ ] **Buttons have aria-labels or visible text**\n  - Accept button: `aria-label=\"Accept order\"` or text \"Accept\"\n  - Ready button: `aria-label=\"Mark order as ready\"`\n  - Complete button: `aria-label=\"Complete order\"`\n  - Cancel button: `aria-label=\"Cancel order\"`\n  - Test: Screen reader announces button purpose\n\n- [ ] **Order cards have semantic role**\n  - Card: `role=\"button\"` (or use <button>)\n  - Label: Order ID as visible content\n  - Test: Screen reader announces \"Order #12847, button\"\n\n- [ ] **Tabs have ARIA roles**\n  - Container: `role=\"tablist\"`\n  - Tab: `role=\"tab\"`\n  - Attribute: `aria-selected=\"true\"` for active tab\n  - Test: Screen reader announces tab state\n\n- [ ] **Modal has ARIA attributes**\n  - Modal: `role=\"dialog\" aria-modal=\"true\"`\n  - Close button: `aria-label=\"Close modal\"`\n  - Test: Screen reader announces dialog, close button\n\n- [ ] **Column headers are semantic**\n  - Use `<h2>` for status headers: \"Received (4)\"\n  - Or use `<div role=\"heading\" aria-level=\"2\">`\n  - Test: Screen reader announces headings\n\n- [ ] **Order details are semantic**\n  - Use semantic HTML: `<h2>`, `<p>`, `<ul>`, `<li>`\n  - Not just `<div>` with styling\n  - Test: Screen reader understands structure\n\n- [ ] **Status badge has aria-label**\n  - Badge: `aria-label=\"2 orders received\"`\n  - Or use: `<span aria-live=\"polite\">(2)</span>`\n  - Test: Screen reader announces count changes\n\n- [ ] **Loading state has aria-label**\n  - Loading button: `aria-label=\"Accepting order, please wait\"`\n  - Or: `<span aria-busy=\"true\" aria-label=\"...\">`\n  - Test: Screen reader announces loading state\n\n- [ ] **Empty state is announced**\n  - Empty column: `<div role=\"status\" aria-live=\"polite\">No orders in this status</div>`\n  - Test: Screen reader announces empty state\n\n#### Implementation Examples\n\n```tsx\n// ✓ Good: Semantic button with aria-label\n<button\n  aria-label=\"Accept order #12847\"\n  onClick={handleAccept}\n>\n  Accept\n</button>\n\n// ✓ Good: Custom button with role and keyboard support\n<div\n  role=\"button\"\n  tabIndex={0}\n  aria-label=\"Order #12847, click to view details\"\n  onKeyPress={(e) => {\n    if (e.key === 'Enter' || e.key === ' ') handleOpen();\n  }}\n>\n  Order #12847\n</div>\n\n// ✓ Good: Order details with semantic HTML\n<article>\n  <h2>Order #12847</h2>\n  <p><strong>Customer:</strong> John Smith</p>\n  <ul>\n    <li>Burger</li>\n    <li>Fries</li>\n  </ul>\n  <p><strong>Status:</strong> Received</p>\n</article>\n\n// ✗ Bad: No label, role, or semantic structure\n<div onClick={handleAccept}>Accept</div>\n<div>12847</div>\n<div>John Smith</div>\n```\n\n---\n\n### 5. Screen Reader Testing\n\n**Requirement:** Application must be usable with screen readers.\n\n#### Testing Procedure\n\n1. **Enable screen reader:**\n   - macOS: VoiceOver (Cmd + F5)\n   - Windows: NVDA (free download) or JAWS\n   - Linux: Orca\n\n2. **Test workflow:**\n   - [ ] Start screen reader\n   - [ ] Navigate to app with Tab key\n   - [ ] Verify headings are announced (\"Kitchen Display System\")\n   - [ ] Verify column headers are announced (\"Received, heading level 2, 4 orders\")\n   - [ ] Verify order cards announce their content (\"Order #12847, John Smith, Burger, Fries, button\")\n   - [ ] Tab to button, verify action (\"Accept button\")\n   - [ ] Press Enter, verify order moves\n   - [ ] Tab to error banner, verify announcement (\"Warning: Connection lost, reconnecting\")\n\n3. **Checklist:**\n   - [ ] All headings are announced\n   - [ ] All button purposes are clear\n   - [ ] Order details are read correctly\n   - [ ] Status changes are announced (via aria-live)\n   - [ ] Empty states are announced\n   - [ ] Modal is recognized as dialog\n   - [ ] Error messages are announced\n\n#### Automated Testing with jest-axe\n\nUse `jest-axe` for automated accessibility scanning in unit tests:\n\n```typescript\nimport { axe, toHaveNoViolations } from 'jest-axe';\n\nexpect.extend(toHaveNoViolations);\n\nit('should have no accessibility violations', async () => {\n  const { container } = render(<OrderCard order={order} />);\n  const results = await axe(container);\n  expect(results).toHaveNoViolations();\n});\n```\n\n---\n\n### 6. Mobile Accessibility\n\n**Requirement:** Touch targets must be at least 44×44 pixels (WCAG 2.5.5 Target Size).\n\n#### Checklist Items\n\n- [ ] **Order cards are at least 44×44px**\n  - Min-height: 140px > 44px ✓\n  - Width: full column width ✓\n  - Test: Measure with DevTools\n\n- [ ] **Buttons are at least 44×44px**\n  - Min-height: 44px ✓\n  - Min-width: 100px (or full-width on mobile)\n  - Test: Measure touch targets\n\n- [ ] **Tab buttons are at least 44×44px**\n  - Min-height: 36px → should be 44px\n  - Increase padding or height\n  - Test: Verify size on mobile device\n\n- [ ] **Spacing between touch targets**\n  - Minimum 8px gap between buttons\n  - Prevents accidental taps\n  - Test: Verify spacing\n\n- [ ] **Modal is full-width on mobile**\n  - Width: 90vw (375px device → 337.5px modal)\n  - Should fit without horizontal scroll\n  - Test: Verify on 375px viewport\n\n#### Implementation\n\n```tsx\n// ✓ Good: 44px touch target\n<button\n  style={{\n    minHeight: '44px',\n    minWidth: '44px',\n    padding: '12px 24px',\n    fontSize: '14px',\n  }}\n>\n  Accept\n</button>\n\n// ✗ Bad: Too small for touch\n<button style={{ padding: '4px 8px', fontSize: '12px' }}>\n  Accept\n</button>\n```\n\n---\n\n### 7. Orientation & Responsive Design\n\n**Requirement:** App must be functional in both portrait and landscape orientations.\n\n#### Checklist Items\n\n- [ ] **Works in portrait (375×667)**\n  - Single column layout\n  - Tab selector visible\n  - No horizontal scroll\n  - Test: Rotate device to portrait\n\n- [ ] **Works in landscape (667×375)**\n  - Still mobile layout (width < 1024px)\n  - Tab selector visible\n  - Adjusted height but same width constraints\n  - Test: Rotate device to landscape\n\n- [ ] **Works on tablet portrait (768×1024)**\n  - Mobile layout (width < 1024px)\n  - Tab selector visible\n  - Larger touch targets more comfortable\n  - Test: View on tablet\n\n- [ ] **Works on tablet landscape (1024×768)**\n  - Desktop layout (5 columns)\n  - No tab selector needed\n  - Columns fit side-by-side\n  - Test: View on tablet landscape\n\n- [ ] **Works on desktop (1920×1080)**\n  - 5-column layout optimal\n  - Large touch targets\n  - Full feature set visible\n  - Test: View on desktop\n\n---\n\n### 8. Color Blindness & Vision\n\n**Requirement:** Information must not rely on color alone.\n\n#### Checklist Items\n\n- [ ] **Status is labeled with text, not just color**\n  - Not: (colored box only)\n  - Yes: \"Status: Received\" (with icon/color optional)\n  - Test: View with color blindness simulator\n\n- [ ] **Order count badges have text**\n  - Tab shows: \"Received (4)\" not just colored badge\n  - Text clearly visible\n  - Test: Verify text is present\n\n- [ ] **Error messages use text + icon**\n  - Error banner: ⚠️ icon + \"Connection lost—reconnecting…\"\n  - Icon is supplementary, not required\n  - Test: Verify message readable without icon\n\n- [ ] **Buttons are labeled, not just colored**\n  - Blue button = \"Accept\" (not just blue)\n  - Red button = \"Cancel\" (not just red)\n  - Text should be visible\n  - Test: Verify button text is clear\n\n- [ ] **Tested with color blindness simulator**\n  - Use: https://www.color-blindness.com/coblis-color-blindness-simulator/\n  - Test with Protanopia (red-blind)\n  - Test with Deuteranopia (green-blind)\n  - Test with Tritanopia (blue-blind)\n  - Verify all information is still clear\n\n---\n\n### 9. Font Size & Readability\n\n**Requirement:** Text must be readable at all zoom levels.\n\n#### Checklist Items\n\n- [ ] **Minimum font size is 11px**\n  - Order ID: 16px ✓\n  - Customer: 14px ✓\n  - Items: 13px ✓\n  - Timestamp: 11px ✓\n  - All text ≥11px\n  - Test: Verify sizes\n\n- [ ] **No horizontal scroll at 200% zoom**\n  - Browser zoom 200%\n  - No horizontal scrollbar\n  - Content reflows vertically\n  - Test: Zoom to 200% in browser\n\n- [ ] **Line height is adequate (≥1.4)**\n  - Spacing between lines should be generous\n  - Default: 1.4 or higher\n  - Test: Verify readability\n\n- [ ] **Text contrast sufficient for low vision**\n  - Large text (18px+): 3:1 contrast\n  - Body text: 4.5:1 contrast\n  - All text exceeds minimums\n  - Test: Use contrast checker\n\n---\n\n### 10. Error Handling & Feedback\n\n**Requirement:** Errors must be clearly communicated to all users.\n\n#### Checklist Items\n\n- [ ] **Error messages are announced by screen readers**\n  - Use `role=\"alert\" aria-live=\"assertive\"`\n  - Message is read immediately\n  - Test: Enable screen reader, trigger error\n\n- [ ] **Error messages are visually clear**\n  - Red icon + text (not just color)\n  - Contrast: #f44336 (red) text on white\n  - Ratio: 5.9:1 ✓\n  - Test: Verify visibility\n\n- [ ] **Error messages have enough time to read**\n  - Toast: visible for ≥4 seconds\n  - Not dismissed too quickly\n  - Test: Measure duration\n\n- [ ] **Success messages are announced**\n  - \"Order #12847 moved to Preparing\"\n  - Use `aria-live=\"polite\"`\n  - Test: Trigger success, verify announcement\n\n---\n\n## Testing Checklist\n\n### Manual Testing\n\n- [ ] Keyboard navigation (Tab, Enter, Escape)\n- [ ] Focus indicators visible on all interactive elements\n- [ ] Screen reader test (VoiceOver/NVDA)\n- [ ] Color contrast verification (WebAIM)\n- [ ] Mobile touch target sizes (44×44px minimum)\n- [ ] Orientation changes (portrait/landscape)\n- [ ] Zoom to 200% (no horizontal scroll)\n- [ ] Color blindness simulation\n\n### Automated Testing\n\n- [ ] jest-axe scans for violations\n- [ ] Unit tests include a11y assertions\n- [ ] Lighthouse accessibility audit (target: 90+)\n\n### Browser & Device Testing\n\n- [ ] Chrome + Screen Reader (VoiceOver on Mac)\n- [ ] Firefox + NVDA (Windows)\n- [ ] Safari + VoiceOver (iOS)\n- [ ] Mobile devices (iPhone, iPad, Android)\n- [ ] Tablets (iPad, Android tablet)\n- [ ] Desktop (Windows, Mac, Linux)\n\n---\n\n## WCAG 2.1 Level AA Conformance Statement\n\nThe KDS application aims to conform to WCAG 2.1 Level AA standards:\n\n✓ **Perceivable:** Text is readable, colors have sufficient contrast, no information relies on color alone  \n✓ **Operable:** Full keyboard navigation, focus management, no keyboard traps, adequate touch targets  \n✓ **Understandable:** Semantic HTML, clear labels, consistent layout, error messages  \n✓ **Robust:** Semantic markup, ARIA labels, compatibility with assistive technologies  \n\n---\n\n## Known Accessibility Issues\n\nNone currently documented.
